# Шаблон для будущих учебных сайтов

Эта папка уже настроена как простой статический учебный сайт:

- `index.html`
- `style.css`
- `script.js`

Ниже зафиксированы правила, по которым теперь всегда нужно собирать такие инструкции.

## Что брать на вход

- основной проект ученика;
- готовый пример или итоговую версию для сравнения;
- картинки и ассеты, если они реально нужны в результате.

Важно: готовые примеры, итоговые версии и технические файлы нужны только нам для анализа. В тексте инструкции для ученика их упоминать не нужно.

## Главное правило по тексту

Инструкция всегда пишется как руководство для ученика, у которого есть только свой проект.

Поэтому в тексте сайта:

- не упоминаем служебные названия вроде `start`, `final`, `update`, `M4L3` и так далее;
- не пишем, что ученик “превращает один файл в другой файл”;
- не ссылаемся на “пример”, “эталон” или “итоговый файл” как на часть текста для ученика;
- обращаемся напрямую: `ты`, `тебе`, `проверь`, `исправь`, `добавь`, `сделай`.

Правильная подача:

- “Добавь кнопку рядом с остальными действиями”
- “Проверь, какой блок у тебя является полем для рисования”
- “Подключи библиотеку перед своим основным JS”

Неправильная подача:

- “Перенеси код из update.js”
- “Сделай как в M4L3”
- “Ученик должен оставить прежние названия файлов”

## Как собирать структуру гайда

1. Сначала сравни проект ученика и готовую версию целиком.
2. Разбей отличия не по файлам, а по смысловым шагам.
3. Составь маршрут так, чтобы ученик шел от простого к сложному.
4. Не делай хаотичный набор карточек. Это должна быть связная учебная дорожка.

Обычный порядок шагов:

1. структура HTML;
2. расположение блоков;
3. рабочая область;
4. сетка;
5. адаптивность;
6. JS-логика;
7. связанные действия вроде очистки, заливки, сохранения;
8. финальная доводка.

Если задача узкая, например только по CSS или только по JS, шаги все равно должны идти от базовой подготовки к проверке результата.

## Формат каждого шага

В каждом шаге должны быть:

- `что было`;
- `что изменить`;
- `зачем`;
- короткая задача;
- 3 основных вопроса;
- 3 уровня подсказок;
- краткая сводка после отметки шага;
- итоговый блок `что было -> что должно получиться`.

После отметки шага как выполненного сайт должен предлагать пройти все вопросы шага для проверки. Только после этого открывается итоговый код.

## Правила для вопросов и подсказок

- Пишем просто и понятно.
- Не задаем мелкие и случайные вопросы, которые не помогают понять задачу.
- Каждый вопрос должен проверять понимание конкретного шага.
- Каждая подсказка должна не просто говорить “что нажать”, а объяснять, почему это работает.

Логика подсказок:

- легкий вопрос открывает маленькую подсказку;
- средний вопрос открывает среднюю подсказку;
- самый точный вопрос открывает почти готовое решение.

Полный код шага показываем только после того, как ученик ответил на все вопросы этого шага.

## Новое обязательное правило по вариантам ответа

Теперь всегда делаем так:

- у каждого вопроса должно быть не 3, а минимум 5 вариантов ответа;
- варианты должны быть достаточно разными, чтобы их нельзя было бездумно перебрать;
- правильный ответ не должен стоять в одном и том же месте;
- варианты нужно перемешивать при рендере шага;
- неправильные варианты должны быть правдоподобными, но не запутанными до абсурда.

Цель не в том, чтобы “подловить” ученика, а в том, чтобы он действительно читал вопрос и думал.

## Новое обязательное правило по ошибкам и таймерам

Теперь всегда делаем защиту от перебора ответов.

Если ученик ошибся в одном и том же вопросе:

- 1 ошибка: блокировка на 5 секунд;
- 2 ошибка: блокировка на 15 секунд;
- 3 ошибка и дальше: блокировка на 30 секунд.

Пока идет блокировка:

- кнопку проверки и варианты ответа временно отключаем;
- показываем уведомление:

`Вариант неправильный, внимательно прочитай задание и попробуй снова. Осталось N секунд.`

Таймер должен обновляться на экране, а после его окончания вопрос снова становится доступен.

## Новое обязательное правило по штрафным вопросам

Если ученик пытается просто перебирать ответы, добавляем усложнение.

Правило:

- на каждые 3 неправильных ответа внутри одного шага добавляется +1 штрафной вопрос;
- штрафные вопросы появляются перед открытием итогового кода этого шага;
- пока штрафные вопросы не пройдены, итоговый код не открывается.

Какими должны быть штрафные вопросы:

- очень простыми;
- по базовым основам HTML, CSS и JS;
- без подвохов;
- короткими и понятными;
- скорее на дисциплину, чем на сложность.

Примеры хороших штрафных вопросов:

- какой тег создает кнопку;
- как обратиться к классу в CSS;
- каким методом взять элемент через `querySelector`;
- какое событие обычно вешают на кнопку;
- какой тег подключает JS;
- какое свойство меняет цвет текста.

## Что важно по логике сайта

- Подсказка открывается только после правильного ответа.
- После отметки шага сайт предлагает пройти все вопросы для закрепления.
- Счетчик шага должен учитывать и обычные, и штрафные вопросы.
- Итоговый код открывается только после полного прохождения вопросов шага.
- Прогресс хранится в `localStorage`.
- В `localStorage` также храним:
  - решенные вопросы;
  - отмеченные шаги;
  - штрафные вопросы;
  - количество ошибок;
  - активные блокировки по времени.
- Должна быть кнопка сброса прогресса.

## Что важно по тону

- Текст должен быть спокойным и человеческим.
- Не давить на ученика.
- Не стыдить за ошибки.
- Даже штрафной механизм должен звучать как часть учебного процесса, а не как наказание.

Правильный тон:

- “Внимательно прочитай задание и попробуй снова”
- “Сначала проверь себя, потом откроется итоговый код”
- “Этот вопрос поможет закрепить базу”

## Проверка перед сдачей

1. Сверь, что все реальные отличия между текущим проектом ученика и нужным результатом покрыты шагами.
2. Проверь, что в ученическом тексте нет служебных имен файлов и названий примеров.
3. Проверь синтаксис `script.js`.
4. Проверь в браузере:
   - шаги строятся;
   - подсказки не открываются без правильного ответа;
   - после ошибки включается таймер;
   - текст с обратным отсчетом обновляется;
   - после 3 ошибок появляется штрафной вопрос;
   - итоговый код открывается только после всех обычных и штрафных вопросов;
   - после отметки шага появляется блок самопроверки;
   - сброс прогресса очищает и основные, и штрафные состояния.
5. Если проект публикуется на GitHub Pages, в `.gitignore` убираем только лишние учебные исходники, а сам сайт и нужные картинки оставляем.

## Короткая формула на будущее

Всегда делаем так:

1. сравниваем проект и нужный результат;
2. пишем инструкцию только про проект ученика;
3. строим понятные шаги;
4. даем 3 вопроса и 3 подсказки на шаг;
5. делаем не меньше 5 вариантов ответа;
6. добавляем таймеры после ошибок;
7. добавляем штрафной вопрос за каждые 3 ошибки;
8. открываем итоговый код только после полной самопроверки.
